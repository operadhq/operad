A team recently shared their experience deploying AI agents for fraud detection at one of the largest banks in the US. Their findings were revealing:

> Claude is a 10x fraud analyst, while still making mistakes every entry-level analyst knows to avoid.

The details paint a familiar picture:

1. **Encyclopedic knowledge, zero institutional awareness.** Claude knows every fraud red flag in the textbook. But it doesn't know what's *normal* at this specific bank. It flags innocuous patterns that any human analyst would recognize as routine.

2. **Powerful analysis, no coordination.** Claude can write SQL, call APIs, and explore data sources you didn't know existed. But it has no concept of case management. Two AI analysts investigating the same case don't know about each other. Coordinating work between them is, in their words, "insanely messy."

3. **Thorough but unaccountable.** Claude will dig through every data source to find evidence. But it will also use legally protected characteristics — like national origin — as the basis for its investigation. There's no audit trail showing *why* it chose that investigative path.

Their solution: better skills. Teach the model what to do and what not to do through improved prompts and tool configurations.

That helps. But it doesn't solve the underlying problem.

## The model is not the bottleneck

Skills tell an agent *how* to think. They're instructions — "don't use protected characteristics," "check with other analysts before proceeding," "learn what's normal for this organization."

But instructions require context. And context requires memory.

Consider: "don't flag this pattern as fraud because it's normal for this bank." Where does the agent learn what's normal? It can't look it up in its training data — that's generic knowledge about fraud, not institutional knowledge about *this specific bank*. Someone has to tell it, and it has to remember.

Now scale that across hundreds of fraud patterns, thousands of cases, and multiple AI analysts working simultaneously. You need:

- A shared knowledge base of what's normal (institutional memory)
- A record of who's investigating what (case coordination)
- An audit trail of every decision and its reasoning (compliance)
- A way to flag when institutional knowledge is outdated (freshness)

These aren't prompt engineering problems. They're infrastructure problems.

## Skills vs. infrastructure

Here's the distinction:

**Skills (prompt engineering):**
```
"When investigating fraud, never use national origin,
race, gender, or other protected characteristics as
evidence or reasoning for your conclusions."
```

This works — until it doesn't. The model might find a proxy variable that correlates with national origin and use that instead. Or it might follow the rule 99% of the time and slip on edge case #347. You're trusting the model to self-regulate based on instructions.

**Infrastructure (decision records):**
```typescript
// Every decision is recorded with full reasoning
await graph.recordDecision({
  selectedAction: 'flag_as_suspicious',
  alternatives: [
    { action: 'mark_as_routine', rejected: 'transaction pattern matches 3 known indicators' }
  ],
  confidence: 0.87,
  reasoning: 'High-value wire transfer to new recipient, velocity spike in last 48 hours'
})

// A behavior automatically audits decisions for compliance
const complianceCheck = behavior({
  name: 'compliance-audit',
  on: ['decision.recorded'],
  handler: async (event, graph, ctx) => {
    const reasoning = event.payload.reasoning as string
    const protectedTerms = ['national origin', 'race', 'gender', 'ethnicity', 'religion']
    const violation = protectedTerms.find(term => reasoning.toLowerCase().includes(term))

    if (violation) {
      await graph.addObject({
        type: 'compliance_flag',
        data: {
          decisionEventId: event.id,
          violation: `Decision reasoning references protected characteristic: ${violation}`,
          severity: 'critical'
        }
      })
    }
  }
})
```

The skill says "don't do this." The infrastructure *catches it when it happens anyway* — and creates a traceable record.

## The coordination problem

The bank team said coordinating work between multiple AI analysts is "insanely messy." Of course it is — the agents have no shared state.

Agent A investigates a suspicious transaction. Agent B investigates a related account. Neither knows the other is working. They might duplicate effort, contradict each other, or miss a connection that only appears when you combine their findings.

With a shared event-sourced graph:

```typescript
// Agent A starts investigating
const case1 = await graph.addObject({
  type: 'investigation',
  data: { transactionId: 'tx_789', assignedTo: 'agent-a', status: 'active' }
})

// Agent B queries before starting — sees Agent A is already on it
const activeInvestigations = await graph.queryObjects({
  type: 'investigation',
  dataMatch: { transactionId: 'tx_789' }
})
// → [{ assignedTo: 'agent-a', status: 'active' }]

// Agent B adds evidence to the existing investigation instead of starting a new one
const evidence = await graph.addObject({
  type: 'evidence',
  data: { source: 'account_history', finding: 'Related account opened 3 days ago' }
})
await graph.addRelation(evidence.id, case1.id, 'relates_to')
```

No special coordination protocol. No message passing. Agents just read and write to the same graph. The event log tracks who did what and when.

## The institutional memory problem

"Labels every innocuous pattern as fraud" — this happens because Claude has generic fraud knowledge but no institutional memory. It doesn't know that this bank's corporate customers regularly make large international wire transfers, or that the insurance division processes high-value claims every quarter.

Skills can partially address this ("our corporate customers often wire >$100k internationally, this is normal"). But skills are static text. Institutional knowledge is dynamic — it changes as the organization changes.

With a graph:

```typescript
// Build institutional baselines over time
await graph.addObject({
  type: 'baseline',
  data: {
    pattern: 'international_wire',
    segment: 'corporate',
    normalRange: { min: 50000, max: 500000 },
    frequency: 'weekly',
    lastVerified: '2026-05-01'
  }
})

// Agent checks baselines before flagging
const baselines = await graph.queryObjects({
  type: 'baseline',
  dataMatch: { pattern: 'international_wire', segment: 'corporate' }
})

// Health tracking flags stale baselines for re-verification
const stale = await graph.getStaleObjects({ thresholdDays: 90 })
// → "Corporate wire baseline was last verified 95 days ago — may need updating"
```

The baselines are objects in the graph. They get verified, updated, and flagged when stale. This is living institutional memory, not a static prompt.

## Complementary, not competing

To be clear: skills and infrastructure aren't either/or. They're complementary:

- **Skills** teach agents *what to do* — investigate fraud patterns, follow compliance rules, use standard workflows
- **Infrastructure** gives agents *what to know* — institutional baselines, case state, decision history, and a way to prove it all

The bank team's skill-based approach is smart. But it's solving half the problem. The other half — persistent memory, shared state, decision provenance, institutional knowledge — requires infrastructure.
